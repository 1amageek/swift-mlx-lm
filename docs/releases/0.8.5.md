# swift-lm 0.8.5

`0.8.5` is a runaway-reasoning containment release. It exposes a new
`GenerationParameters.maxReasoningTokens` knob that callers can use to bound
how far a thinking model is allowed to wander inside its reasoning channel
before it must close `</think>`.

## What 0.8.5 Changes

### `GenerationParameters.maxReasoningTokens`

Until now, `GenerationParameters.maxTokens` only bounded the **visible**
answer. When `reasoning == .separate` and the model exposes a thinking-tag
policy, the runtime quietly raised the raw cap to a generous multiplier
(`max(maxTokens * 16, maxTokens + 256)`) so reasoning had room to finish.
That escape hatch let degenerate reasoning loops run far past the caller's
intended budget — e.g. a `maxTokens: 256` request could legally generate
~4096 raw tokens before being cut off.

`maxReasoningTokens` is the explicit knob that makes those loops fail fast:

```swift
let parameters = GenerationParameters(
    maxTokens: 256,            // visible answer cap
    maxReasoningTokens: 512,   // hard cap on reasoning channel
    reasoning: .separate
)
// total raw cap is now exactly maxTokens + maxReasoningTokens = 768
```

Behavior:

- `nil` (default) preserves the historical generous multiplier — existing
  callers see no change.
- `0` collapses the raw cap to `maxTokens`, refusing to spend any token on
  reasoning.
- Any explicit value sets the raw cap to `maxTokens + maxReasoningTokens`,
  even if smaller than the historical floor of `maxTokens + 256`.
- The knob has no effect when `reasoning != .separate` or when no
  thinking-tag policy is present — there is nothing to cap.

### Internals: `maxRawTokenCount` is now testable

`LanguageModelContext.maxRawTokenCount(forVisibleLimit:maxReasoningTokens:visibilityPolicy:reasoningVisibility:)`
is promoted from `private` to `internal static` so its contract can be
pinned by direct unit tests instead of only being exercised through
end-to-end generation. The new `MaxRawTokenCountTests` suite covers the
nine relevant cases (no policy / inline / separate × nil / 0 / explicit /
explicit-below-floor / explicit-above-multiplier).

## Correctness

Validated before release:

- `MaxRawTokenCountTests` (10 tests) pass — pins the cap function across
  every mode.
- Existing thinking E2E paths continue to pass with `maxReasoningTokens:
  nil`, confirming default callers are unaffected.
- End-to-end check via jardis CLI on Gemma 4 E2B (transcript-channel
  reasoning) and Qwen 3.5 0.8B (inline `<think>` markup) with
  `--show-think`: reasoning and final answer are both produced and
  cleanly separated.

```bash
xcodebuild test-without-building -scheme swift-lm-Package -destination 'platform=macOS' \
  -only-testing:'SwiftLMTests/MaxRawTokenCountTests' \
  -parallel-testing-enabled NO
```

## Notes

- No migration required. `GenerationParameters.init` gained
  `maxReasoningTokens: Int? = nil` as a defaulted parameter; existing
  call sites compile and behave identically.
- Backends that bridge their own configuration into `GenerationParameters`
  (e.g. `MetalConfiguration` in `AnyFoundationModels`) need a one-line
  passthrough to expose the knob to their callers — see
  `AnyFoundationModels` 0.5.6.

dependencies: [
