# swift-lm 0.7.1

`0.7.1` unblocks MLX VLM bundle loading by canonicalizing tensor names
at STAF conversion time. The MLX convention for VLM text backbones
(`language_model.model.*`) now resolves to the canonical HF form
(`model.language_model.*`) once, inside the converter, rather than
requiring every downstream consumer to know both layouts.

## What 0.7.1 Changes

### MLX VLM tensor name canonicalization

MLX publishes VLM text backbones under `language_model.model.*` while
HuggingFace â€” and the rest of swift-lm â€” expects `model.language_model.*`.
Previously this mismatch prevented MLX VLM bundles (e.g.
`mlx-community/Qwen3.5-0.8B-MLX-bf16`, `mlx-community/Qwen3.5-0.8B-4bit`)
from loading.

The fix is a single rewrite pass at the STAF conversion boundary:

- `TensorNameCanonicalizer.detect(from:)` picks the rule set by
  inspecting safetensors header names once â€” O(1) per source convention,
  not per tensor.
- `STAFConversionEntry` now carries both the canonical `name` and the
  original `sourceName`, so companion `.scales` / `.biases` lookups
  still resolve against the on-disk safetensors layout while everything
  downstream (kernel dispatch, parameter resolution, KV cache setup)
  sees the canonical form only.
- STAF format version bumped `1 â†’ 2` to invalidate stale caches from
  earlier releases that were written with the MLX-form names.
- `BenchmarkSupport` now re-validates existing STAF files via
  `STAFConverter().isValid()` instead of trusting file existence, so a
  format-version mismatch triggers a clean re-conversion rather than a
  load-time crash.

### Diagnostic probe for Qwen3.5 Q4 decode plan

`Qwen35Q4DecodeProbeTests.dumpDecodeKernels` prints the Q4 Qwen3.5
decode plan (kernel names, layer indices, weight tensor names) for the
first 90 steps. Useful for confirming the Q4 BatchedProjection
decomposition fires on DeltaNet SSM layers and the right GEMV variants
are dispatched per projection. Diagnostic only (no assertions), same
pattern as `Gemma4Q4HiddenProbeTests` / `Gemma4Q4WeightProbeTests`.

## Scope

- Qwen3.5 BF16 text generation from `mlx-community/Qwen3.5-0.8B-MLX-bf16`
  now loads and generates healthy tokens (30â€“31 / 31 unique tokens on
  the agreement harness).
- Qwen3.5 Q4 from `mlx-community/Qwen3.5-0.8B-4bit` loads successfully,
  but the Q4 decode path through Qwen3.5's DeltaNet/SSM quantization
  collapses to token 0. This is tracked as a known issue in
  `Qwen35Q4AgreementTests` via `withKnownIssue` and is separate from
  the canonicalization work in this release.
- Gemma4 Q4 / Q6 / Q8 paths are unaffected (Attention-only, no DeltaNet).

## Upgrade notes

- Existing STAF caches written by 0.7.0 or earlier will be re-generated
  automatically on first load after upgrade, due to the format version
  bump. This is a one-time cost.
- No public API changes.
