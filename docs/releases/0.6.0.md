# swift-lm 0.6.0

`0.6.0` introduces automatic cross-component kernel fusion, adds EmbeddingGemma
as a first-class declaration, fixes the LFM2 decode regression caused by kernel
name aliasing, and pins a ground-truth catalog of which quantization schemes
are actually validated per model family.

## What 0.6.0 Includes

### Automatic Kernel Fusion

- `FusionContract` / `FusionSynthesizer` / `SynthesizedFragment` / `KernelScaffold`
  replace hand-written fused kernels with compiler-driven synthesis
- `MetalEntryCollector.fuseCrossComponent` greedily fuses adjacent fusable
  fragments based on contract compatibility alone (parallelism, ports,
  threadgroup memory), with no fragment-specific switch/case in the compiler
- LoopGroup architecture splits synthesis into register-local vs
  threadgroup-staged phases, enabling sandwich-norm fusion
  (`RMSNorm + ResidualAdd + Copy + RMSNorm`) and per-row loop adaptation for
  fragments with mixed parallelism patterns
- `fusionContract` + `kernelBody` are now complete for all fusable fragments
  (Reduction, ElementwiseFragment, ScalarMultiplyFragment, QKNormFragment,
  RoPEFragment, ResidualAddFragment, CopyFragment)

### Models

- EmbeddingGemma declaration (`mlx-community/embeddinggemma-300m-bf16` and
  `mlx-community/embeddinggemma-300m-4bit`) with full retrieval/embedding path
- Vision encoder components: `PatchEmbedding`, `PositionEmbedding`, `Pooling`,
  `Standardize`, with correctness fixes for Gemma4 and Qwen3.5 vision paths
- Direct quantized GEMM prefill path and GPU-side embedding post-processing

### Correctness Fixes

- **LFM2 decode regression** (`5fbd879e`): `SynthesizedFragment.kernelName` now
  encodes a composition tag derived from fragment types. Before, distinct
  compositions such as `CopyFragment+Reduction` and
  `ResidualAddFragment+Reduction` both resolved to
  `synthesized_2way_4p_row2048_f32_wbf16`, letting
  `MetalKernelSourceCatalog.generatedNames` and `SharedPipelineCache` alias two
  different MSL bodies under one name and silently corrupt decode output
- **Gemma4 4-way fusion weight aliasing** (`7c4a37f7`): disambiguate
  `weightRole` at emission so fused sandwich-norm kernels never share a weight
  binding across distinct operations
- **BF16 fusion precision**: align fused kernel intermediate precision across
  both backends so vision encoder outputs match reference
- **Silent fallback in sampling gate** (`717eb55b`): `topK` / `minP` /
  `presencePenalty` now fail loudly when requested but unsupported instead of
  silently reverting to argmax

### Supported Quantization Catalog

- New `docs/design/supported-quantizations.md` is the ground-truth catalog for
  which weight formats and KV cache schemes are actually validated per model
  family (Gemma4, EmbeddingGemma, LFM2, Qwen3.5), complementing the
  architectural direction in `docs/design/quantization.md`
- `SupportedQuantizationsContractTests` pins
  `QuantizationSchemeIdentifier` classification into three disjoint sets
  (registered weight formats, declared-but-unsupported, KV-cache-only) so
  future scheme additions require a coordinated update to the document

### Refactors

- `Sources/Models` reorganized into per-model subdirectories
- `SwiftLM/QwenVision` renamed to `SwiftLM/Qwen35` for family naming
  consistency
- `WeightNamingConvention` refactored from enum dispatch to protocol
- `MetalCompiler` restructured around capability protocols; dead code removed
- Tests split into generic vs model-specific subdirectories; broken excluded
  tests deleted

### Documentation

- Developer guide for adding new `ModelComponent`s and Fragments
- `CLAUDE.md` / `AGENTS.md` aligned with the automatic-fusion implementation
- EmbeddingGemma benchmark recorded: swift-lm BF16 **66.2 emb/s** vs MLX
  62.0 emb/s (+6.8%)

## Supported Scope

- Apple Silicon with a usable Metal device
- Platforms currently declared in `Package.swift`: macOS 26+, iOS 26+,
  visionOS 26+
- Text and chat generation through `ModelInput`
- Text embeddings through `TextEmbeddingInput`
- Prompt snapshot reuse through `PromptSnapshot`
- Multimodal execution for the documented Qwen3.5 and Gemma4 paths

## Not Included in 0.6.0

- Tool calling or structured function-calling APIs
- Non-Metal execution backends
- Training or fine-tuning workflows
- Broad cross-platform validation outside the package's declared Apple
  platform targets
